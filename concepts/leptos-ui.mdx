Oxichrome uses [Leptos](https://leptos.dev) to render the popup and options page UIs for your browser extension. Leptos is a fine-grained reactive Rust framework: instead of re-rendering a virtual DOM tree, it updates only the exact DOM nodes that depend on changed state. This makes it fast, predictable, and a natural fit for WebAssembly — and it lets you write your entire frontend in Rust without touching JavaScript or HTML files directly. This page explains how `#[oxichrome::popup]` and `#[oxichrome::options_page]` wire Leptos into the extension, and covers the key patterns you will use in practice.

## How the popup and options page are mounted

When you annotate a function with `#[oxichrome::popup]`, the macro does two things:

1. Wraps your function with `#[leptos::component]`, which registers it as a Leptos component.
2. Emits a `#[wasm_bindgen]` export `__oxichrome_mount_popup()` that calls `leptos::mount::mount_to_body(YourComponent)`.

The generated `popup.js` shim loads the WASM module, then immediately calls this entry point:

```javascript
// Auto-generated by oxichrome. Do not edit.
import init, { __oxichrome_mount_popup } from './wasm/my_extension.js';

try {
    await init();
    __oxichrome_mount_popup();
} catch (e) {
    console.error('[oxichrome] Popup init failed:', e);
    document.body.textContent = 'Error: ' + e.message;
}
```

`#[oxichrome::options_page]` follows the same pattern, calling `__oxichrome_mount_options()` instead.

<Note>
The mount handle returned by `mount_to_body` is leaked with `std::mem::forget`. This is intentional — the reactive tree must live for the entire lifetime of the popup page, so it must not be dropped.
</Note>

## Reactive state with `RwSignal`

`RwSignal<T>` is a read-write reactive value. Any part of the `view!` template that reads a signal will automatically re-render when that signal changes.

```rust
let count = RwSignal::new(0i32);

// Read in the template — re-evaluates whenever count changes
view! {
    <div>{move || count.get()}</div>
}
```

To update a signal, call `.set()` or `.update()`:

```rust
count.set(42);
count.update(|c| *c += 1);
```

To read a signal's current value without registering a reactive dependency (useful inside event handlers or async blocks), use `.get_untracked()`:

```rust
let val = count.get_untracked();
```

## Loading state on mount with `Effect::new`

`Effect::new` runs a closure reactively after the component is mounted. Use it to load initial values from storage or other async sources.

```rust
Effect::new(move || {
    wasm_bindgen_futures::spawn_local(async move {
        if let Ok(Some(val)) = oxichrome::storage::get::<i32>("counter").await {
            count.set(val);
        }
    });
});
```

<Note>
`Effect::new` runs after the initial render. Signals updated inside the effect trigger a second render, which is how the displayed value updates once the storage read completes.
</Note>

## Async operations with `spawn_local`

All Chrome API calls in Oxichrome are async. Inside a Leptos component — which is synchronous — you drive async work using `wasm_bindgen_futures::spawn_local`. This spawns a future on the WASM microtask queue without blocking the current reactive context.

```rust
let save = move |_| {
    let val = count.get_untracked();
    wasm_bindgen_futures::spawn_local(async move {
        let _ = oxichrome::storage::set("counter", &val).await;
    });
};
```

## Templates with the `view!` macro

The `view!` macro accepts JSX-like HTML syntax and returns `impl IntoView`. Text nodes must be wrapped in double quotes. Event handlers use the `on:event=handler` syntax.

```rust
view! {
    <div class="controls">
        <button on:click=decrement>"-"</button>
        <span>{move || count.get()}</span>
        <button on:click=increment>"+"</button>
    </div>
}
```

Dynamic content is expressed as closures that capture signals:

```rust
// Re-evaluates whenever count changes
<div>{move || count.get()}</div>

// Conditional rendering
{move || (count.get() > 0).then(|| view! { <span>"Positive"</span> })}
```

## Inline styles with `<style>` blocks

You can write CSS directly inside the `view!` macro using a `<style>` element. Oxichrome's generated `popup.html` and `options.html` shells have minimal base styles (`margin: 0`, `min-width: 240px`), so your component styles take full control of the layout.

```rust
view! {
    <style>
        "body { background: #0f0f0f; color: #e0e0e0; font-family: sans-serif; }"
    </style>
    <h1>"My Extension"</h1>
}
```

<Tip>
Because the popup renders inside a Chrome extension page (an isolated origin), there is no stylesheet inheritance from the surrounding website. You own the entire document.
</Tip>

## Complete example: the counter popup

The following is the full popup component from the counter extension. It demonstrates all of the patterns above working together: `RwSignal` for reactive state, `Effect::new` for loading from storage on mount, `spawn_local` for async saves on each button click, and inline styles via a `<style>` block.

```rust src/lib.rs
#[oxichrome::popup]
fn Popup() -> impl IntoView {
    let count = RwSignal::new(0i32);

    // Load the persisted counter value on mount
    Effect::new(move || {
        wasm_bindgen_futures::spawn_local(async move {
            if let Ok(Some(val)) = oxichrome::storage::get::<i32>("counter").await {
                count.set(val);
            }
        });
    });

    // Helper: update the signal and persist to storage
    let update = move |delta: i32| {
        count.update(|c| *c += delta);
        let val = count.get_untracked();
        wasm_bindgen_futures::spawn_local(async move {
            let _ = oxichrome::storage::set("counter", &val).await;
        });
    };

    let increment = move |_| update(1);
    let decrement = move |_| update(-1);
    let reset = move |_| {
        count.set(0);
        wasm_bindgen_futures::spawn_local(async move {
            let _ = oxichrome::storage::set("counter", &0i32).await;
        });
    };

    view! {
        <style>
            "* { margin: 0; padding: 0; box-sizing: border-box; }
            body { width: 240px; font-family: -apple-system, BlinkMacSystemFont, sans-serif;
                   background: #0f0f0f; color: #e0e0e0; padding: 1.25rem; }
            h1 { font-size: 0.65rem; font-weight: 500; letter-spacing: 0.15em;
                 text-transform: uppercase; color: #888; margin-bottom: 1rem; text-align: center; }
            .count { font-size: 3.5rem; font-weight: 200; font-variant-numeric: tabular-nums;
                     line-height: 1; text-align: center; color: #fff; margin-bottom: 1.25rem; }
            .controls { display: flex; gap: 0.5rem; justify-content: center; }
            button { width: 44px; height: 44px; border-radius: 50%; border: 1.5px solid #333;
                     background: transparent; color: #e0e0e0; font-size: 1.25rem; cursor: pointer; }
            .reset { width: auto; border-radius: 22px; padding: 0 1rem;
                     font-size: 0.7rem; text-transform: uppercase; }"
        </style>
        <h1>"Counter"</h1>
        <div class="count">{move || count.get()}</div>
        <div class="controls">
            <button on:click=decrement>"\u{2212}"</button>
            <button on:click=increment>"+"</button>
            <button class="reset" on:click=reset>"Reset"</button>
        </div>
    }
}
```

## Patterns summary

<AccordionGroup>
  <Accordion title="Reading a signal in the template">
    Wrap the read in a `move ||` closure so Leptos can track the dependency and update only that DOM node when the value changes.

    ```rust
    <span>{move || my_signal.get()}</span>
    ```
  </Accordion>
  <Accordion title="Writing to a signal from an event handler">
    Event handlers are synchronous closures. Write to the signal directly; `spawn_local` for any async follow-up work.

    ```rust
    let on_click = move |_| {
        my_signal.set(new_value);
        wasm_bindgen_futures::spawn_local(async move {
            let _ = oxichrome::storage::set("key", &new_value).await;
        });
    };
    ```
  </Accordion>
  <Accordion title="Loading data on mount">
    Use `Effect::new` to run code after the component renders. Because effects re-run when their reactive dependencies change, use `.get_untracked()` inside effects to avoid infinite loops.

    ```rust
    Effect::new(move || {
        wasm_bindgen_futures::spawn_local(async move {
            if let Ok(Some(v)) = oxichrome::storage::get::<MyType>("key").await {
                my_signal.set(v);
            }
        });
    });
    ```
  </Accordion>
  <Accordion title="Conditional rendering">
    Return an `Option<impl IntoView>` from a reactive closure. When the option is `None`, nothing is rendered.

    ```rust
    {move || error.get().map(|msg| view! { <p class="error">{msg}</p> })}
    ```
  </Accordion>
  <Accordion title="Rendering a list">
    Map over a `Vec` inside a reactive closure and call `.collect_view()` at the end.

    ```rust
    {move || items.get().into_iter().map(|item| view! {
        <li>{item}</li>
    }).collect_view()}
    ```
  </Accordion>
</AccordionGroup>
