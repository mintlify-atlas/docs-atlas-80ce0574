When you run `cargo oxichrome build`, the CLI performs a precise sequence of steps that turn your annotated Rust source into a directory you can load directly in Chrome or Firefox — no configuration files to write and no JavaScript to maintain. This page walks through every stage of the pipeline, the output structure it produces, and the differences between Chromium and Firefox builds.

## Pipeline stages

<Steps>
  <Step title="Verify the wasm32-unknown-unknown target">
    The CLI runs `rustup target list --installed` and checks whether `wasm32-unknown-unknown` is present. If it is not, it runs `rustup target add wasm32-unknown-unknown` automatically before proceeding.
  </Step>
  <Step title="Verify wasm-bindgen-cli version">
    The CLI reads your `Cargo.lock` to find the exact version of the `wasm-bindgen` crate your project uses. It then checks whether `wasm-bindgen --version` matches. If the version does not match, or if `wasm-bindgen` is not installed at all, the CLI runs `cargo install wasm-bindgen-cli --version=<version>` to install the correct version.

    <Note>
    This version-pinning step prevents the `wasm-bindgen` ABI mismatch error that occurs when the CLI version differs from the crate version.
    </Note>
  </Step>
  <Step title="Compile the crate to WebAssembly">
    The CLI runs:

    ```bash
    cargo build --lib --target wasm32-unknown-unknown
    ```

    For a release build (`cargo oxichrome build --release`), it adds `--release`:

    ```bash
    cargo build --lib --target wasm32-unknown-unknown --release
    ```

    The compiled artifact is written to `target/wasm32-unknown-unknown/{debug|release}/{crate_name}.wasm`.
  </Step>
  <Step title="Run wasm-bindgen">
    The CLI calls `wasm-bindgen` on the `.wasm` file:

    ```bash
    wasm-bindgen --target web --out-dir dist/{browser}/wasm/ --out-name {crate_name} {wasm_file}
    ```

    This produces two files in `dist/{browser}/wasm/`:
    - `{crate_name}.js` — the JavaScript glue module that imports and initializes the WASM module.
    - `{crate_name}_bg.wasm` — the stripped WebAssembly binary.

    Hyphens in crate names are replaced with underscores (e.g., `counter-extension` becomes `counter_extension`).
  </Step>
  <Step title="Parse src/lib.rs for metadata">
    The CLI parses `src/lib.rs` using `syn` without recompiling, walking the AST to detect which macros are present and extract their arguments:

    - `#[oxichrome::extension(...)]` — reads `name`, `version`, `description`, `permissions`
    - `#[oxichrome::background]` — records each background function name
    - `#[oxichrome::on(...)]` — records each event handler name and its namespace/event
    - `#[oxichrome::popup]` — sets `has_popup = true`
    - `#[oxichrome::options_page]` — sets `has_options_page = true`
    - `#[oxichrome::content_script(...)]` — records each content script's function name, match patterns, `run_at`, `all_frames`, and `css` files

    This metadata drives every subsequent step.
  </Step>
  <Step title="Generate manifest.json">
    The CLI serializes the detected metadata into a Manifest V3 JSON file at `dist/{browser}/manifest.json`. The file always includes:

    - `manifest_version: 3`
    - `name`, `version`, `description` from `#[oxichrome::extension]`
    - `permissions` from `#[oxichrome::extension]`
    - `background.service_worker: "background.js"` (Chromium) or `background.scripts: ["background.js"]` (Firefox)
    - `content_security_policy` set to allow WebAssembly via `wasm-unsafe-eval`
    - `web_accessible_resources` exposing `wasm/*` to `<all_urls>` (required for content scripts to load WASM)

    Conditional fields are only written when the corresponding macro is detected:

    - `action.default_popup` is written only when `#[oxichrome::popup]` is present
    - `options_ui` is written only when `#[oxichrome::options_page]` is present
    - Each `#[oxichrome::content_script]` produces an entry in the `content_scripts` array
  </Step>
  <Step title="Generate JS shims">
    The CLI generates all JavaScript files needed to bootstrap the WASM module. None of these files need to be written manually.

    **`background.js`** — always generated. Imports `init` and each background/event-handler wrapper from the `wasm/` module, awaits `init()`, calls event-handler registrations first, then calls background functions:

    ```javascript
    // Auto-generated by oxichrome. Do not edit.
    import init, { __oxichrome_bg_start, __oxichrome_register_handle_install }
        from './wasm/counter_extension.js';

    async function start() {
        try {
            await init();
            __oxichrome_register_handle_install();
            __oxichrome_bg_start();
            console.log('[oxichrome] Background service worker initialized.');
        } catch (e) {
            console.error('[oxichrome] Failed to initialize WASM:', e);
        }
    }

    start();
    ```

    **`popup.html` + `popup.js`** — generated when `#[oxichrome::popup]` is detected. The HTML shell loads `popup.js` as an ES module; the JS file awaits `init()` then calls `__oxichrome_mount_popup()`.

    **`options.html` + `options.js`** — generated when `#[oxichrome::options_page]` is detected. Same pattern as popup.

    **`content_script_{fn_name}.js`** — generated for each `#[oxichrome::content_script]` function. Because content scripts cannot use static `import` statements, the shim uses `chrome.runtime.getURL` to build a dynamic import path:

    ```javascript
    // Auto-generated by oxichrome. Do not edit.
    // Content script: inject
    (async () => {
        try {
            const src = chrome.runtime.getURL('wasm/counter_extension.js');
            const wasm = await import(src);
            await wasm.default();
            wasm.__oxichrome_cs_inject();
            console.log('[oxichrome] Content script "inject" initialized.');
        } catch (e) {
            console.error('[oxichrome] Content script "inject" failed to initialize:', e);
        }
    })();
    ```
  </Step>
  <Step title="Copy static assets">
    If a `static/` directory exists in the crate root, its contents are copied recursively to `dist/{browser}/`. This is where you place icons, fonts, or CSS files referenced by content scripts. See the [static assets guide](/guides/static-assets) for details.
  </Step>
  <Step title="Optimize with wasm-opt (optional)">
    If `wasm-opt` is found in `PATH`, the CLI runs:

    ```bash
    wasm-opt -Oz dist/{browser}/wasm/{crate_name}_bg.wasm -o dist/{browser}/wasm/{crate_name}_bg.wasm
    ```

    The `-Oz` flag optimizes aggressively for size. If `wasm-opt` is not installed, this step is skipped with a warning — it is not required for the extension to function.

    <Tip>
    Install `wasm-opt` from the [binaryen](https://github.com/WebAssembly/binaryen) project (`brew install binaryen` on macOS, or download from the GitHub releases page) to reduce your extension's WASM binary size significantly.
    </Tip>
  </Step>
</Steps>

## Output structure

After a successful build, `dist/chromium/` (or `dist/firefox/`) contains:

```
dist/
└── chromium/
    ├── manifest.json
    ├── background.js
    ├── popup.html              # only if #[oxichrome::popup] present
    ├── popup.js                # only if #[oxichrome::popup] present
    ├── options.html            # only if #[oxichrome::options_page] present
    ├── options.js              # only if #[oxichrome::options_page] present
    ├── content_script_inject.js  # one per #[oxichrome::content_script]
    ├── wasm/
    │   ├── counter_extension.js
    │   └── counter_extension_bg.wasm
    └── icon.png                # copied from static/ if present
```

## Chromium vs. Firefox differences

The only differences between the two output directories are in `manifest.json`. The JavaScript shims and WASM files are identical.

<Tabs>
  <Tab title="Chromium manifest.json">
    ```json
    {
      "manifest_version": 3,
      "name": "Counter Extension",
      "version": "0.1.0",
      "background": {
        "service_worker": "background.js",
        "type": "module"
      }
    }
    ```

    Chromium uses a service worker model: `background.service_worker` points to the entry script.
  </Tab>
  <Tab title="Firefox manifest.json">
    ```json
    {
      "manifest_version": 3,
      "name": "Counter Extension",
      "version": "0.1.0",
      "background": {
        "scripts": ["background.js"],
        "type": "module"
      },
      "browser_specific_settings": {
        "gecko": {
          "id": "counter-extension@oxichrome.dev"
        }
      }
    }
    ```

    Firefox uses `background.scripts` instead of `background.service_worker`, and requires a `browser_specific_settings.gecko.id`. Oxichrome generates the ID automatically from the extension name.
  </Tab>
</Tabs>

To build for Firefox, pass `--target firefox`:

```bash
cargo oxichrome build --target firefox
```

See the [Firefox guide](/guides/firefox) for loading the extension in `about:debugging`.
